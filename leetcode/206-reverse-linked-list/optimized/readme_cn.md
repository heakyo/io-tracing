## 问题描述

给定一个单链表的头节点 `head`，请你反转这个链表，然后返回反转后的链表头节点。

想象你手里有一串珠子，穿在一根线上。现在要求你把珠子的顺序倒过来——
第一颗变最后一颗，最后一颗变第一颗。链表反转做的就是这件事，只不过珠子是
节点，线是指针。

**示例：**

```
输入:  1 -> 2 -> 3 -> 4 -> 5 -> NULL
输出:  5 -> 4 -> 3 -> 2 -> 1 -> NULL
```

---

## 核心思路

和原版一样，这里使用 **O(n) 迭代反转**——一次遍历，逐个掰转箭头方向。
但优化版把变量命名理顺了，让每个变量都只干一件事：

| 变量 | 用途 |
|------|------|
| `prev` | 指向「已经构建好的反转链」的头部，初始为 `NULL` |
| `next` | 临时保存「当前节点的下一个节点」，防止断链后找不到路 |
| `head` | **直接用作游标**，不再额外声明 `p`；每轮循环后向前推进 |

原版用了三个命名指针（`rh`、`p`，加上复用 `head` 做临时变量），语义含混。
优化版只需要两个命名变量（`prev`、`next`），`head` 本身就是游标，干干净净。

循环体只有 4 行，像四步舞步一样循环往复：

```c
next = head->next;   /* 第 1 步：抄下下一站的地址 */
head->next = prev;   /* 第 2 步：把当前节点的箭头掰向「已反转链」 */
prev = head;         /* 第 3 步：当前节点加入「已反转链」 */
head = next;         /* 第 4 步：游标前进到下一个节点 */
```

完整函数：

```c
struct ListNode* reverseList(struct ListNode* head)
{
	struct ListNode *prev = NULL, *next;

	while (head) {
		next = head->next;
		head->next = prev;
		prev = head;
		head = next;
	}

	return prev;
}
```

## 逐步推演

以链表 `[1, 2, 3, 4, 5]` 为例，逐步观察三个变量的状态变化。

**初始状态：**

```
prev = NULL
head -> [1] -> [2] -> [3] -> [4] -> [5] -> NULL
```

| 步骤 | 操作前 head | next = head->next | head->next = prev | prev = head | head = next | 操作后 prev 链 |
|:----:|:-----------:|:-----------------:|:------------------:|:-----------:|:-----------:|:--------------:|
| 1 | `[1]` | `[2]` | `[1] -> NULL` | `[1]` | `[2]` | `1 -> NULL` |
| 2 | `[2]` | `[3]` | `[2] -> [1]` | `[2]` | `[3]` | `2 -> 1 -> NULL` |
| 3 | `[3]` | `[4]` | `[3] -> [2]` | `[3]` | `[4]` | `3 -> 2 -> 1 -> NULL` |
| 4 | `[4]` | `[5]` | `[4] -> [3]` | `[4]` | `[5]` | `4 -> 3 -> 2 -> 1 -> NULL` |
| 5 | `[5]` | `NULL` | `[5] -> [4]` | `[5]` | `NULL` | `5 -> 4 -> 3 -> 2 -> 1 -> NULL` |

**循环结束：** `head == NULL`，退出 `while`。返回 `prev`，即 `[5]`。

用费曼的话说：每一步你做的事情就是——把当前节点从旧链上「摘」下来，「贴」到
新链的最前面。就像发扑克牌时把每张牌扣在牌堆顶上，最后翻开牌堆，顺序自然
就反了。

## 易错点分析

### (a) 为什么 `head` 可以直接当游标？

C 语言中，函数参数是**按值传递**的。调用 `reverseList(myList)` 时，函数内部
的 `head` 是 `myList` 的一份**副本**——一个独立的局部变量。你在函数里随便
修改 `head` 的值（让它往前走），完全不会影响调用者手里的 `myList`。

所以，`head` 天生就可以当游标用，不需要再声明一个多余的 `p`。

### (b) 操作顺序不能乱

```c
next = head->next;    /* ← 必须第一步！ */
head->next = prev;    /* 这一步会断开 head 和原链的连接 */
```

如果你先执行 `head->next = prev`，原来的 `head->next`（也就是链表的下一个
节点）就丢了，再也找不回来。所以**保存 next 必须在断链之前**。

这就好比你拆一节火车车厢之前，必须先记住后面那节车厢在哪儿，否则整列火车
就断成了两截，后半截不知去向。

### (c) 命名优化带来的好处

| 原版 | 优化版 | 改进 |
|------|--------|------|
| `rh`（reversed head） | `prev` | 名字直接表达「前一个节点」，无需注释即可理解 |
| 复用 `head` 做临时变量 | `next` | 专用变量保存下一个节点，职责单一，不易混淆 |
| 额外声明 `p` 做游标 | 直接用 `head` | 减少一个变量，代码更短，逻辑更直白 |

好的命名就是最好的注释。当每个变量名都在说「我是谁、我干什么」的时候，
读代码就像读故事一样流畅。

## 复杂度分析

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| **时间** | **O(n)** | 遍历链表一次，n 为节点数 |
| **空间** | **O(1)** | 只用了 `prev` 和 `next` 两个额外指针，与链表长度无关 |

与原版完全相同——优化**纯粹在代码清晰度上**，不改变算法本身的时空开销。

---

## 总结

优化版本以相同的 **O(n) 时间、O(1) 空间**性能实现链表反转，但代码更加简洁
明了：

1. **2 个命名变量而非 3 个**——`prev` 和 `next`，各司其职。
2. **直接使用 `head` 作为游标**——消除了冗余的 `p` 指针，因为函数参数本身
   就是一份局部副本，天然可做迭代变量。
3. **`next` 取代了对 `head` 参数的复用**——使代码意图一目了然：`next` 就是
   「下一个节点」，不需要在脑中做变量角色切换。

用费曼的话来总结：**如果你不能用简单的方式说清楚，说明你还没真正理解。**
这个优化版本，就是把「真正理解」写进了代码里——每一行都在说人话，每个变量
都只讲一个故事。
