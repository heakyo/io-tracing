## 问题描述

给定一个整数数组 `nums`，找出所有满足以下条件的三元组 `[nums[i], nums[j], nums[k]]`：

- 三个下标互不相同：`i != j`、`i != k`、`j != k`
- 三数之和为零：`nums[i] + nums[j] + nums[k] == 0`

返回的结果中**不能包含重复的三元组**。

用一个最直白的比喻来理解这道题：想象你有一堆标了正负数字的积木块，你需要从中挑出三块，使它们上面的数字加起来刚好等于零。而且，同样组合的三块积木只能算一次。

**示例：**

```
输入：nums = [-1, 0, 1, 2, -1, -4]
输出：[[-1, -1, 2], [-1, 0, 1]]
```

---

## 核心思路

这份代码采用了一种"**由外向内夹逼**"的策略，与标准的"固定一个元素 + 双指针"方法不同。让我们像讲故事一样，一步步拆解作者的思路：

### 第一步：排序

和大多数双指针方案一样，首先对数组进行升序排序。排序后，负数在左侧，正数在右侧，零在中间。这为后续的搜索提供了有序的基础。

```c
qsort(nums, numsSize, sizeof(int), compare);
```

### 第二步：初始化外层双指针

设置两个指针 `left = 0`（指向最小元素）和 `right = numsSize - 1`（指向最大元素）。这两个指针构成三元组的**外层一对**，然后在它们之间搜索第三个元素 `k`。

```c
left = 0;
right = numsSize - 1;
```

### 第三步：计算外层和，分情况搜索中间元素

每次循环计算 `val = nums[left] + nums[right]`，然后根据 `val` 的正负号决定搜索策略：

- **当 `val < 0` 时**（外层和为负）：需要一个正数来补偿。从 `right - 1` 开始**向左**搜索，但只在 `nums[k] > 0` 的范围内搜索。
- **当 `val > 0` 时**（外层和为正）：需要一个负数来补偿。从 `left + 1` 开始**向右**搜索，但只在 `nums[k] < 0` 的范围内搜索。
- **当 `val == 0` 时**：需要找到一个零。在 `left + 1` 到 `right - 1` 之间线性搜索值为 `0` 的元素。

### 第四步：启发式指针推进

找到（或未找到）三元组后，代码用一段启发式逻辑决定移动 `left` 还是 `right`：

- 在 `val < 0` 的分支中，对 `k` 做 `++k`，检查 `val + nums[k]` 的符号来决定推进哪个指针。
- 在 `val > 0` 的分支中，对 `k` 做 `--k`，同样检查符号来决策。
- 在 `val == 0` 的分支中，根据 `k` 在区间中的相对位置来选择移动方向。

推进时会跳过相邻的重复元素，以避免产生重复三元组。

## 逐步推演

以 LeetCode 示例 `[-1, 0, 1, 2, -1, -4]` 为例。

**排序后**：`[-4, -1, -1, 0, 1, 2]`，下标 `0` 到 `5`。

```
下标:    0    1    2    3    4    5
值:     -4   -1   -1    0    1    2
```

---

### 迭代 1

| 项目       | 值                                    |
|:-----------|:--------------------------------------|
| left       | 0 (nums[0] = -4)                     |
| right      | 5 (nums[5] = 2)                      |
| val        | -4 + 2 = **-2**                      |
| 分支       | val < 0，需要找 -val = 2              |
| 搜索方向   | 从 k = right-1 = 4 向左，条件 nums[k]>0 |

搜索过程：
- k=4：nums[4] = 1 > 0，val + 1 = -1 ≠ 0，继续
- k=3：nums[3] = 0，不满足 nums[k] > 0，**循环终止**

**结果**：未找到，found = 0。

**指针推进**（val < 0 分支）：此时 k=3，执行 `++k` → k=4，检查 val + nums[4] = -2 + 1 = -1 < 0，不满足 `> 0`，走 else 分支：保存 k=left=0，跳过所有等于 nums[0]=-4 的元素，left 变为 **1**。

---

### 迭代 2

| 项目       | 值                                    |
|:-----------|:--------------------------------------|
| left       | 1 (nums[1] = -1)                     |
| right      | 5 (nums[5] = 2)                      |
| val        | -1 + 2 = **1**                       |
| 分支       | val > 0，需要找 -val = -1             |
| 搜索方向   | 从 k = left+1 = 2 向右，条件 nums[k]<0 |

搜索过程：
- k=2：nums[2] = -1 < 0，val + (-1) = 0 == 0 ✓

**结果**：找到三元组 **[-1, -1, 2]** ✓

**指针推进**（val > 0 分支）：执行 `--k` → k=1，检查 val + nums[1] = 1 + (-1) = 0，不满足 `< 0`，走 else 分支：保存 k=right=5，跳过所有等于 nums[5]=2 的元素，right 变为 **4**。

---

### 迭代 3

| 项目       | 值                                    |
|:-----------|:--------------------------------------|
| left       | 1 (nums[1] = -1)                     |
| right      | 4 (nums[4] = 1)                      |
| val        | -1 + 1 = **0**                       |
| 分支       | val == 0，搜索中间的零                 |
| 搜索方向   | k 从 2 到 3，找 nums[k] == 0          |

搜索过程：
- k=2：nums[2] = -1 ≠ 0，继续
- k=3：nums[3] = 0 == 0 ✓

**结果**：找到三元组 **[-1, 0, 1]** ✓

**指针推进**（val == 0 分支）：k=3，区间长度 right-left = 3，中点 = 1.5，k=3 > 1.5，走 else 分支：保存 k=left=1，跳过所有等于 nums[1]=-1 的元素。left 跳过下标 1 和 2（都是 -1），left 变为 **3**。

---

### 迭代 4

left = 3，right = 4。检查 left < right → 3 < 4 成立。

| 项目       | 值                                    |
|:-----------|:--------------------------------------|
| left       | 3 (nums[3] = 0)                      |
| right      | 4 (nums[4] = 1)                      |
| val        | 0 + 1 = **1**                        |
| 分支       | val > 0，需要找 -val = -1             |
| 搜索方向   | 从 k = left+1 = 4 向右，条件 nums[k]<0 |

搜索过程：
- k=4：nums[4] = 1，不满足 nums[k] < 0，**循环立即终止**

**结果**：未找到，found = 0。

**指针推进**（val > 0 分支）：执行 `--k` → k=3，检查 val + nums[3] = 1 + 0 = 1，不满足 `< 0`，走 else 分支：保存 k=right=4，跳过等于 nums[4]=1 的元素，right 变为 **3**。

---

### 循环终止

left = 3，right = 3。条件 left < right 不成立，退出循环。

**最终输出**：`[[-1, -1, 2], [-1, 0, 1]]` ✓

> **注意**：在这个特定的例子中，算法碰巧给出了正确答案。但在许多其他输入下（如 `[-3,-1,0,1,2,4]`），它会遗漏三元组。后文的"易错点分析"将详细说明原因。

## 易错点分析

这份代码的思路很有创意，但存在几个关键缺陷，导致它无法通过所有测试用例。

### (a) 内层搜索受符号限制

当 `val < 0` 时，内层循环只在 `nums[k] > 0` 的元素中搜索；当 `val > 0` 时，只在 `nums[k] < 0` 的元素中搜索。

乍一看这似乎合理——负的 `val` 需要一个正数来补偿，对吧？但问题在于：**目标值 `-val` 本身不一定落在那个符号范围里**。

例如，假设 `val = -1`，那么需要找 `nums[k] = 1`。如果恰好只有 `nums[k] = 1` 这一个正数元素，循环可以找到。但如果 `val = -2` 而我们需要的 `nums[k] = 2` 恰好在零的左边（比如数组中有多个零），这个搜索就会在找到它之前因为碰到 `nums[k] <= 0` 而提前终止。

更关键的是，这种按符号划分搜索范围的做法，**完全遗漏了 `nums[k] == 0` 但 `val != 0` 的情况**。比如 `val = 0` 只在 `val == 0` 分支处理，但如果 left 和 right 指向的值恰好互为相反数且非零，我们需要的第三个元素是 0，但代码走的是 `val < 0` 或 `val > 0` 分支，永远不会去找零。

等等——如果 `nums[left] = -2` 且 `nums[right] = 2`，则 `val = 0`，会走 `val == 0` 分支，确实会搜索零。但如果 `nums[left] = -3` 且 `nums[right] = 3`，同理 `val = 0` 也没问题。所以真正的问题出在**边界条件**：例如 `val = -1` 时需要找 1，搜索从 `right - 1` 开始向左走，但如果中间有零挡住了搜索路径，循环会因 `nums[k] <= 0` 而停止。

### (b) 指针推进基于启发式

这是最致命的缺陷。找到（或未找到）一个三元组后，代码不是系统地枚举所有可能的 `(left, right)` 配对，而是根据**临近元素的值**来"猜测"应该移动哪个指针。

这种启发式会**跳过有效的配置**。考虑输入 `[-3, -1, 0, 1, 2, 4]`：

```
排序后: [-3, -1, 0, 1, 2, 4]
期望输出: [[-3,-1,4], [-3,1,2], [-1,0,1]]
```

算法首先设 left=0（-3），right=5（4），val=-3+4=1>0，在负数中搜索 -1，找到 nums[1]=-1。于是记录三元组 `[-3, -1, 4]`。

接下来执行指针推进逻辑：`--k` → k=0，`val + nums[0] = 1 + (-3) = -2 < 0` 成立，走 if 分支：保存 k=left=0，跳过所有等于 nums[0]=-3 的元素，**left 移到 1**。

从此刻起，left 已经跳过了 -3 这个元素。三元组 `[-3, 1, 2]` 需要 left=0 指向 -3，但 left 再也不会回到 0 了。这个合法的三元组被永久遗漏。

### (c) 提前返回值不一致

```c
if (nums[0] > 0 || nums[numsSize - 1] < 0) {
	*returnSize = 0;
	return NULL;
}
```

正常路径返回 `malloc` 分配的 `output` 指针，但提前返回路径返回 `NULL`。如果调用者统一用 `free(output)` 释放内存，对 `NULL` 调用 `free` 虽然在 C 标准中是安全的，但代码在提前返回前还做了一次多余的 `*returnColumnSizes = malloc(...)`（第 72 行已经分配过一次），这导致第一次分配的内存泄漏。

此外，正常路径和异常路径的返回类型语义不一致，给调用者带来了处理上的困惑。

### (d) 内层搜索缺少边界检查

以 `val < 0` 分支为例：

```c
k = right - 1;
while (nums[k] > 0) {
	if (val + nums[k] == 0) {
		found = 1;
		break;
	}
	k--;
}
```

这个循环没有显式检查 `k > left` 的条件。它依赖的是"排序数组中正数在右侧，k 递减终会碰到非正数从而退出"这个隐式假设。但如果整个数组都是正数（虽然前面有提前返回，但如果提前返回逻辑被修改），`k` 可能递减到负下标，造成越界访问。

即使在当前代码中，`k` 也没有保证停在 `left + 1` 之上。如果 left 和 right 之间全是正数，`k` 会一路递减到 `left` 甚至更左，导致找到的 `k` 可能等于 `left`（违反了 `i != k` 的约束）。

## 复杂度分析

### 时间复杂度：O(n²)

- **排序**：`qsort` 为 O(n log n)。
- **外层 while 循环**：每次迭代 `left` 增加或 `right` 减少（至少一个），因此最多执行 O(n) 次。
- **内层搜索**：每次最多扫描 left 到 right 之间的所有元素，即 O(n)。
- **总计**：O(n log n) + O(n) × O(n) = **O(n²)**。

### 空间复杂度：O(1) 额外空间

- 除了存储输出结果的数组外，算法只使用了 `left`、`right`、`k`、`val`、`found` 等常数个变量。
- 输出数组的空间不计入额外空间复杂度（这是题目要求返回的）。

> **重要提醒**：虽然复杂度看起来合理，但**算法的结果是不正确的**。在多种输入下，它会遗漏有效的三元组。一个时间复杂度正确但结果错误的算法是没有意义的。

## 总结

这份代码尝试了一种与众不同的解法：**固定两个外部元素（最小和最大），然后在中间搜索第三个元素**。这种"由外向内"的视角确实很有创意——它试图利用排序后数组的正负分布来加速搜索。

然而，这个方法存在根本性的缺陷：

1. **受限的内层搜索**按符号过滤候选元素，导致遗漏符号不匹配的有效三元组。
2. **启发式指针推进**没有系统地遍历所有 `(left, right)` 配对，导致有效配置被跳过。
3. **返回值不一致**和**缺少边界检查**带来了工程层面的隐患。

标准的解法是**固定一个元素，然后用双指针在剩余元素中搜索**：

```c
for (i = 0; i < n - 2; i++) {
	// 跳过重复的 i
	left = i + 1;
	right = n - 1;
	while (left < right) {
		sum = nums[i] + nums[left] + nums[right];
		if (sum == 0) {
			// 记录结果，跳过重复
			left++;
			right--;
		} else if (sum < 0) {
			left++;
		} else {
			right--;
		}
	}
}
```

这种方法之所以可靠，是因为它**系统地枚举了第一个元素的所有可能**，然后对每个固定的第一个元素，双指针**完整地扫描剩余区间**。不存在"猜测该移动哪个指针"的问题——移动规则完全由 `sum` 的正负决定，简洁且正确。

> 费曼曾说："如果你不能把它简单地解释出来，说明你还没有真正理解它。"这份代码的复杂启发式逻辑恰恰说明了一个道理：当一个算法需要太多特殊情况处理时，也许应该退一步，寻找更简洁的方案。
