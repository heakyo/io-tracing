## 题目描述

有一个单链表，给你其中某个节点的指针（**不是头节点的指针**），请你删除该节点。

注意：你拿到的只是"要被删除的那个节点本身"，而不是链表的头。你没有办法
访问该节点之前的任何节点。

**约束条件：**

- 给定的节点**不是尾节点**（保证它后面至少还有一个节点）
- 链表中至少有两个节点
- 链表中所有节点的值互不相同

**示例：**

```
输入：head = [4, 5, 1, 9]，node = 5
输出：[4, 1, 9]
```

## 核心思路

> 想象教室里同学们按顺序坐成一排，老师要求"删掉第二个人"。
> 但你不能把椅子搬走（因为你够不到前面的人来重新接线）。
> 怎么办？让第三个人把名字写到第二个人的名牌上，然后第三个人离开。
> 从外面看，第二个人"消失"了——其实是第二个人**变成了**第三个人，
> 然后把真正的第三个人踢出队列。

算法只有两步：

1. **复制值**：把下一个节点的值复制到当前节点。
2. **跳过下一个节点**：让当前节点的 `next` 指向下下个节点。

这样，当前节点在逻辑上就"变成了"下一个节点，而原来的下一个节点被架空、
从链表中脱离。

核心代码如下：

```c
void deleteNode(struct ListNode* node)
{
	node->val = node->next->val;
	node->next = node->next->next;
}
```

用费曼的话说：**不是删掉自己，而是把自己变成下一个人，再把下一个人踢掉。**

## 逐步推演

以 `[4, 5, 1, 9]` 删除节点 `5` 为例，手动追踪每一步。

**初始状态：**

```
head --> [4] -> [5] -> [1] -> [9] -> NULL
                 ^
                node（要删除的节点）
```

**第 1 步：`node->val = node->next->val`**

把 `node->next`（即节点 `1`）的值复制到 `node`（即节点 `5`）上：

```
head --> [4] -> [1] -> [1] -> [9] -> NULL
                 ^      ^
                node   node->next（值已被复制过来）
```

节点 `5` 的值从 `5` 变成了 `1`。现在链表中出现了两个值为 `1` 的节点。

**第 2 步：`node->next = node->next->next`**

让 `node->next` 跳过原来的下一个节点，直接指向下下个节点（值为 `9`）：

```
head --> [4] -> [1] ---------> [9] -> NULL
                 ^
                node

                        [1] （被跳过，脱离链表）
```

**最终结果：** `[4, 1, 9]`

被跳过的那个节点（原来的 `1` 节点）仍然存在于内存中，但已经不在链表里了。

## 易错点分析

### (a) 内存泄漏

原版代码执行 `node->next = node->next->next` 后，被跳过的那个节点不再有
任何指针指向它，但它的内存没有被 `free`。在 LeetCode 的在线评测环境中这
通常不会报错，但在实际工程中这是一个内存泄漏。

修正方法：

```c
void deleteNode(struct ListNode* node)
{
	struct ListNode *tmp = node->next;  /* 先保存要删除的节点 */
	node->val = tmp->val;
	node->next = tmp->next;
	free(tmp);                          /* 释放被跳过的节点 */
}
```

### (b) 为何不能删除尾节点

如果 `node` 是尾节点，那么 `node->next` 为 `NULL`：

- `node->next->val` 会对空指针解引用，导致段错误（segfault）。
- 即使我们能把 `node->val` 设为某个特殊值，也无法让前一个节点的 `next`
  指向 `NULL`，因为我们根本访问不到前一个节点。

所以题目保证给定的节点不是尾节点。

### (c) 这不是"真正"删除该节点

从内存地址的角度看，被删掉的其实是 `node->next` 那个节点（它的内存被释放
或脱离链表），而 `node` 本身的内存地址没有变，只是值被覆盖了。

如果外部有指针指向原来的 `node->next`，在操作后该指针会变成悬空指针。如果
外部代码依赖节点的内存地址做映射（比如哈希表以节点地址为键），这种"复制+
跳过"的手法可能导致逻辑错误。

### (d) 不能用"常规删除"

常规链表删除需要访问被删节点的**前驱节点**，将前驱的 `next` 指向被删节点的
`next`。但本题不提供头节点，无法遍历到前驱，所以只能用"复制值+跳过"这种
变通手法。

## 总结

本题的核心技巧是**复制+跳过**：因为无法访问前驱节点，所以不能用常规方法
断开当前节点，只能让当前节点"伪装"成下一个节点（复制其值），然后将真正的
下一个节点从链表中移除（跳过其指针）。整个操作只需两行代码，时间和空间
复杂度均为 O(1)。需要注意的是，这种做法存在内存泄漏（被跳过的节点未释放）、
不能处理尾节点、且删除的并非原节点本身而是其后继节点——理解这些局限性，
才算真正掌握了这道题。
