## 1. 题目描述

给定一个链表的头节点 `head` 和一个非负整数 `k`，将链表向右旋转 `k` 个位置。

**示例 1:**

```
输入: head = [1,2,3,4,5], k = 2
输出: [4,5,1,2,3]
```

**示例 2:**

```
输入: head = [0,1,2], k = 4
输出: [2,0,1]
```

**约束条件:**

- 链表节点数范围 `[0, 500]`
- `-100 <= Node.val <= 100`
- `0 <= k <= 2 * 10^9`

"向右旋转 k 个位置" 的含义：将链表末尾的 k 个节点搬到链表头部。当 k 大于链表
长度时，等价于 `k % size` 次旋转。

## 2. 核心思路 — 成环再断开

原版算法虽然正确，但使用了不必要的 dummy 节点，变量名也容易引起误解（`prev`
实际上是 tail，`next` 声明过早但很晚才使用）。优化版用更直观的方式完成同样的
操作：**先把链表首尾相连形成环，再在正确位置断开**。

整个算法分三步：

**第一步：遍历链表，计算长度，找到尾节点。**

从 `head` 出发，用 `tail` 指针走到链表末尾。计数从 1 开始，循环结束时 `size`
就是节点总数，`tail` 指向最后一个节点。

**第二步：成环。**

执行 `tail->next = head`，将尾节点的 next 指回头节点，链表变成一个环。

**第三步：找到断开位置，断开环。**

从 `head` 出发，向前走 `size - n - 1` 步（其中 `n = k % size`），到达
`new_tail`。`new_tail->next` 就是新的头节点。将 `new_tail->next` 置为 NULL，
环断开，旋转完成。

```c
struct ListNode* rotateRight(struct ListNode* head, int k)
{
	struct ListNode *tail, *new_tail;
	int size, n, i;

	if (head == NULL || head->next == NULL || k == 0)
		return head;

	/* 第一步: 计算长度, 找到尾节点 */
	size = 1;
	tail = head;
	while (tail->next) {
		size++;
		tail = tail->next;
	}

	n = k % size;
	if (n == 0)
		return head;

	/* 第二步: 成环 */
	tail->next = head;

	/* 第三步: 走到断开点, 断开环 */
	new_tail = head;
	for (i = 0; i < size - n - 1; i++)
		new_tail = new_tail->next;

	head = new_tail->next;
	new_tail->next = NULL;

	return head;
}
```

为什么走 `size - n - 1` 步？向右旋转 n 位，等价于把最后 n 个节点搬到前面。
断开点在从头数第 `size - n` 个节点之后（0-indexed 为第 `size - n - 1` 个）。
从 `head` 出发走 `size - n - 1` 步正好到达该位置。

## 3. 逐步推演

以 `[1, 2, 3, 4, 5]`，`k = 2` 为例。

### 3.1 初始状态

```
head
 |
[1] -> [2] -> [3] -> [4] -> [5] -> NULL
```

### 3.2 第一步：遍历，计算 size，找到 tail

`size` 从 1 开始，`tail` 从 `head` 出发。每次 `tail->next` 非空，`size++`，
`tail` 前进一步。

| 迭代 | tail 指向 | size |
|------|-----------|------|
| 初始 | 1         | 1    |
| 1    | 2         | 2    |
| 2    | 3         | 3    |
| 3    | 4         | 4    |
| 4    | 5         | 5    |

循环结束：`size = 5`，`tail` 指向节点 5。

计算有效旋转次数：`n = k % size = 2 % 5 = 2`。`n != 0`，继续执行。

### 3.3 第二步：成环

执行 `tail->next = head`：

```
head                          tail
 |                             |
[1] -> [2] -> [3] -> [4] -> [5] --+
 ^                                 |
 +---------------------------------+
```

此时链表形成一个环：`5 -> 1 -> 2 -> 3 -> 4 -> 5 -> 1 -> ...`

### 3.4 第三步：找到断开点，断开环

需要走的步数：`size - n - 1 = 5 - 2 - 1 = 2`。

`new_tail` 从 `head`（节点 1）出发：

| 迭代 | i | new_tail 指向 |
|------|---|---------------|
| 初始 | - | 1             |
| 1    | 0 | 2             |
| 2    | 1 | 3             |

循环结束：`new_tail` 指向节点 3。

执行断开操作：

```
head = new_tail->next;   // head 指向节点 4
new_tail->next = NULL;   // 节点 3 的 next 置为 NULL
```

### 3.5 最终结果

```
head
 |
[4] -> [5] -> [1] -> [2] -> [3] -> NULL
```

输出：`[4, 5, 1, 2, 3]`，符合预期。

## 5. 易错点分析

### 5.1 成环安全性

成环操作（`tail->next = head`）本身是安全的，但**必须在之后断开**。如果在成环
后提前返回或发生异常，链表将永远保持环形结构，后续任何遍历操作都会陷入死循环。

本算法的安全性由代码结构保证：成环之后只有三行代码（一个 for 循环和两条赋值），
不存在提前返回的路径。两个提前返回点（`head == NULL` 和 `n == 0`）都在成环
操作之前。

### 5.2 walk 的 off-by-one

走 `size - n - 1` 步是最容易出错的地方。差一步就会导致断开位置偏移，结果完全
错误。

理解这个数字的关键：

- 向右旋转 n 位 = 最后 n 个节点搬到前面
- 新尾节点是从头数第 `size - n` 个节点（1-indexed）
- 从 `head` 出发到第 `size - n` 个节点需要走 `size - n - 1` 步

用具体数字验证：`size = 5`，`n = 2`。新尾节点应该是第 3 个节点（值为 3）。
从 `head`（节点 1）走 `5 - 2 - 1 = 2` 步：1 -> 2 -> 3。正确。

**常见错误：**

| 错误写法                 | 后果                     |
|--------------------------|--------------------------|
| `i < size - n`           | 多走一步，断在第 4 个节点 |
| `i < size - n - 2`       | 少走一步，断在第 2 个节点 |
| `i <= size - n - 1`      | 等价于 `i < size - n`    |

### 5.3 k 为 0 或 k 为 size 的整数倍

当 `k == 0` 时，直接在开头的边界判断返回，不进入任何计算。

当 `k` 是 `size` 的整数倍时，`n = k % size = 0`，在第二个判断点返回。此时
链表不需要任何变动。这两个判断必须在成环操作之前，否则会成环而不断开。

### 5.4 空链表与单节点链表

`head == NULL`：无节点可旋转，直接返回。

`head->next == NULL`：只有一个节点，旋转任意次结果不变，直接返回。这个判断
也避免了后续 `size = 1`、`n = k % 1 = 0` 导致的不必要计算。

## 6. 与原版对比

### 6.1 逻辑对比表

| 方面           | 原版                             | 优化版                          |
|----------------|----------------------------------|---------------------------------|
| dummy 节点     | 使用 `dummy = {0, head}`         | 无需 dummy 节点                 |
| 计数起始       | `size = 0`，循环内 `size++`      | `size = 1`，循环内 `size++`     |
| 找尾节点       | `prev` 在循环中跟踪尾节点       | `tail` 直接命名为尾节点         |
| 连接操作       | `prev->next = head`（成环）      | `tail->next = head`（成环）     |
| 断开操作       | `cur->next = NULL`               | `new_tail->next = NULL`         |
| 返回值         | `dummy.next = next; return ...`  | `head = new_tail->next; return` |
| 变量数量       | 5 个（cur, next, prev, dummy, i）| 4 个（tail, new_tail, size, i） |
| 概念模型       | 隐式成环 + dummy 间接返回        | 显式成环再断开                  |

### 6.2 原版问题详解

**问题一：变量名误导。** 原版用 `prev` 追踪尾节点。`prev` 的常见含义是"当前
节点的前一个节点"，但在这里它在循环结束后指向链表最后一个节点，实质是 `tail`。
读代码时需要心理翻译，增加认知负担。

**问题二：dummy 节点多余。** 原版声明 `dummy = {0, head}` 并通过 `dummy.next`
返回结果。但旋转链表不存在"头节点可能被删除"的场景（所有节点都保留，只是顺序
变化），dummy 节点在这里没有实际作用。最终 `dummy.next = next` 只是一个间接
赋值，直接 `return next` 或 `return new_tail->next` 效果完全一样。

**问题三：`next` 变量声明过早。** 原版在函数开头声明 `next`，但直到断开操作
时才使用（`next = cur->next`）。在长函数中，声明和使用距离越远，读者越难追踪
变量的用途。优化版根本不需要这个中间变量。

**问题四：概念不显式。** 原版的 `prev->next = head` 实际上也是在成环，但代码
没有明确表达"先成环再断开"这个意图。优化版通过注释和变量命名让读者立即理解
算法的两个阶段。

## 7. 复杂度分析

### 时间复杂度：O(n)

| 阶段                 | 操作次数       |
|----------------------|----------------|
| 遍历计算 size        | n 次（精确）   |
| 走到断开点           | size - n - 1 次 |
| 成环和断开           | O(1)           |
| **总计**             | **< 2n，即 O(n)** |

其中 n 为链表节点数。最坏情况下（`k % size = 1`），第二阶段走 `size - 2` 步，
总操作约 `2n - 2` 次。

### 空间复杂度：O(1)

只使用了固定数量的指针变量（`tail`、`new_tail`）和整数变量（`size`、`n`、`i`），
不随输入规模增长。

与原版相比，复杂度完全相同。优化版的改进在于代码可读性，而非渐进性能。

## 8. 函数参考

以下为 `optimized/main.c` 中所有函数的说明。

### rotateRight

```c
struct ListNode* rotateRight(struct ListNode* head, int k);
```

核心算法函数。将以 `head` 为头的链表向右旋转 `k` 个位置，返回新的头节点。
使用"成环再断开"方法，详见第 2 节。

### insert_list

```c
struct ListNode* insert_list(int *nums, int size);
```

从整数数组构建链表。从数组末尾向前遍历，依次在链表头部插入节点，保证链表顺序
与数组一致。返回链表头节点。

### show_list

```c
void show_list(char *type, struct ListNode* head);
```

打印链表内容。`type` 为标签字符串（如 `"Input "` 或 `"Output"`），用于区分
输出的上下文。格式为 `type: [ val1 val2 ... ]`。

### free_list

```c
static void free_list(struct ListNode *head);
```

释放链表所有节点的内存。从头到尾逐个释放，防止内存泄漏。

### check_list

```c
static int check_list(struct ListNode *head, int *expected, int len);
```

验证链表内容是否与期望数组一致。逐节点比较值，同时检查长度是否匹配。返回 1
表示通过，0 表示不匹配。

### run_test

```c
static void run_test(const char *name, int *nums, int len,
                     int k, int *expected, int expected_len);
```

运行单个测试用例。流程：构建链表 -> 打印输入 -> 调用 `rotateRight` -> 打印
输出 -> 用 `assert` 验证结果 -> 释放内存。`name` 为测试用例名称，失败时
`assert` 会终止程序并报告位置。

## 9. 总结

优化版的核心改进不在于算法复杂度（两者均为 O(n) 时间、O(1) 空间），而在于
**代码表达与思维模型的一致性**。

"成环再断开"是旋转链表最自然的心理模型：把链表想象成一条首尾相连的环形链条，
然后在正确的位置剪断，就得到了旋转后的结果。优化版的代码直接映射了这个模型——
`tail->next = head` 是成环，`new_tail->next = NULL` 是断开，变量名 `tail`
和 `new_tail` 精确描述了它们的角色。

原版虽然逻辑上也在做同样的事情（`prev->next = head` 本质也是成环），但被
dummy 节点、误导性的变量名和间接的返回方式掩盖了意图。读者需要在脑中重新
构建算法模型，而不是直接从代码中读出来。

好的代码不只是正确的代码，而是让读者不需要额外推理就能理解意图的代码。
