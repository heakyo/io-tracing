## 问题描述

给定链表的头节点 `head`，判断链表中是否有环。

如果链表中某个节点的 `next` 指针再次指向了前面已经出现过的节点，那么链表中就存在环。

如果存在环返回 `true`，否则返回 `false`。

---

## 核心思路

想象一条环形跑道上有两个人：一个走得慢（乌龟），一个走得快（兔子）。如果跑道是直的，兔子会先跑到终点；但如果跑道有环，兔子会绕回来，最终和乌龟相遇。

这就是经典的 **Floyd 龟兔赛跑算法**。具体做法：

1. 两个指针 `slow` 和 `fast` **都从 `head` 出发**。
2. 每轮循环里，**先前进**（`slow` 走一步，`fast` 走两步），**再检查**是否相遇。
3. 如果 `fast` 走到了 `NULL`（跑道到头了），说明没有环。
4. 如果 `slow == fast`（两人相遇了），说明有环。

```c
bool hasCycle(struct ListNode *head)
{
	struct ListNode *slow = head, *fast = head;

	while (fast && fast->next) {
		slow = slow->next;
		fast = fast->next->next;

		if (slow == fast)
			return true;
	}

	return false;
}
```

与原版相比，优化版更清晰：

- **(a) 标准命名**：`slow`/`fast` 替代 `sp`/`qp`，一眼就知道谁快谁慢。
- **(b) 同一起点**：两者都从 `head` 开始，而非相隔一步。
- **(c) 先前进后比较**：这是教科书标准模式，避免了起点处的误判。
- **(d) 无冗余保护**：不需要 `head->next == NULL` 的提前返回，`while` 条件已全部覆盖。
- **(e) 代码更少**：无单独的初始化块，结构一目了然。

## 逐步推演

以 `[3, 2, 0, -4]`，`pos = 1` 为例。

节点编号与连接关系：

```
Node0(3) -> Node1(2) -> Node2(0) -> Node3(-4)
               ^                        |
               +------------------------+
                     （环：Node3 -> Node1）
```

| 轮次 | 操作前 slow | 操作前 fast | slow 前进 1 步 | fast 前进 2 步 | slow == fast? |
|:----:|:-----------:|:-----------:|:--------------:|:--------------:|:-------------:|
| 初始 | Node0(3)    | Node0(3)    | —              | —              | （不比较）    |
| 1    | Node0(3)    | Node0(3)    | Node1(2)       | Node2(0)       | 否            |
| 2    | Node1(2)    | Node2(0)    | Node2(0)       | Node1(2)       | 否            |
| 3    | Node2(0)    | Node1(2)    | Node3(-4)      | Node3(-4)      | **是！**      |

第 3 轮结束时 `slow == fast`，返回 `true`。链表有环。

注意第 2 轮中 `fast` 从 Node2 走到 Node3 再到 Node1——它绕过了环的入口。这正是快指针"套圈"的过程。

## 易错点分析

### (a) 为什么先前进再比较？

两个指针都从 `head` 出发。如果**先比较再前进**，那第一次比较时 `slow == fast == head`，永远返回 `true`——即使链表根本没有环。

所以必须**先走一步，再检查是否相遇**。这是"同一起点"方案的配套要求。

### (b) `while` 条件为什么是 `fast && fast->next`？

`fast` 每次走两步：`fast = fast->next->next`。

- 如果 `fast` 本身为 `NULL`，访问 `fast->next` 就是段错误。
- 如果 `fast->next` 为 `NULL`，访问 `fast->next->next` 就是段错误。

所以必须同时检查 `fast != NULL` **且** `fast->next != NULL`，缺一不可。

### (c) 为什么 `fast` 一定能追上 `slow`？

假设环的长度为 $C$。当 `slow` 进入环时，`fast` 已经在环里。设此时两者距离为 $d$（$0 < d < C$）。

每走一步，`fast` 比 `slow` 多走 1 步，两者的距离**缩小 1**。经过 $d$ 步后距离变为 0，即相遇。

所以最多走 $C$ 步（环长度）就一定相遇，不可能"擦肩而过"。

### (d) 空链表怎么办？

当 `head` 为 `NULL` 时，`fast = head = NULL`。`while` 条件 `fast && fast->next` 立即为假，直接返回 `false`。

不需要任何特殊处理，这就是优化版省去 `head->next == NULL` 检查的原因。

## 复杂度分析

| 维度 | 复杂度 | 说明 |
|:----:|:------:|:-----|
| 时间 | $O(n)$ | 无环时 `fast` 走完整个链表；有环时最多绕环一圈即相遇。总步数与节点数线性相关。 |
| 空间 | $O(1)$ | 只用了 `slow` 和 `fast` 两个指针，不随输入规模增长。 |

与原版完全相同——优化版的改进纯粹在**代码清晰度**，算法本质没有变化。

---

## 总结

优化版使用教科书式的 Floyd 龟兔赛跑算法：

1. **同一起点**：`slow` 和 `fast` 都从 `head` 出发，逻辑对称。
2. **先前进后比较**：避免起点误判，符合标准写法。
3. **标准命名**：`slow`/`fast` 直观明了。
4. **无冗余保护**：`while` 条件覆盖了空链表、单节点等所有边界情况。

结果与原版相同，但代码更简洁、结构更清晰——真正做到了"一眼看懂"。
