## 为什么要优化？

原始版本使用嵌套循环和 `found` 标志，逻辑比必要的更冗长。外层 `while` 遍历 `s`，内层 `while` 在 `t` 中寻找当前字符，找到后设置 `found = true` 并跳出内层循环；找不到则立刻返回 `false`。这样写当然正确，但需要维护两个指针 `p1`、`p2`，一个布尔标志 `found`，以及一次提前返回检查——对于一个本质上只需要"扫一遍 `t`"的问题来说，机械零件太多了。

优化版反转了视角：不是"对 `s` 的每个字符扫描 `t`"，而是"扫描 `t` 一遍，每当 `t` 匹配 `s` 的当前字符就推进 `s`"。这一视角转换消除了 `found` 标志、嵌套循环和提前返回检查——整个函数压缩到 3 行：

```c
bool isSubsequence(char *s, char *t)
{
	for (; *t; t++)
		if (*t == *s)
			s++;
	return !*s;
}
```

想象你拿着一张购物清单（`s`）走过超市货架（`t`）。你不会对清单上每一项都从头逛一遍超市——你只走一趟，看到清单上的东西就划掉，走到头看看清单是否全部划完。优化版做的正是这件事。

---

## 核心思路

对 `t` 进行单遍扫描。每当 `t` 的当前字符匹配 `s` 的当前字符时，推进 `s` 指针。当 `t` 耗尽后，如果 `s` 也被完全消耗（即 `*s == '\0'`，等价于 `!*s` 为真），则 `s` 是 `t` 的子序列。

指针 `s` 在此充当**进度标记**——它只在发生匹配时前进，而 `t` 每一步都前进。这天然地保持了字符的相对顺序：`s` 中靠后的字符只可能匹配 `t` 中更靠后的位置，因为 `t` 的扫描从不回头。

## ASCII 流程图

```
          开始
           |
           v
      +----------+
      | *t != 0? |
      +----------+
        |       |
       是      否
        |       |
        v       v
  +-----------+  +----------+
  | *t == *s? |  | return   |
  +-----------+  | !*s      |
    |       |    +----------+
   是      否
    |       |
    v       |
  +-----+   |
  | s++ |   |
  +-----+   |
    |       |
    v       v
  +-----+
  | t++ |
  +-----+
    |
    +---> (回到 *t != 0? 判断)
```

说明：`for (; *t; t++)` 控制外层循环——只要 `*t` 非零就继续。循环体内只有一个 `if`：当 `*t == *s` 时执行 `s++`。无论是否匹配，`t` 都会通过 `for` 语句的递增部分自动前进。循环结束后，`return !*s` 一句判定结果。

## 逐步演练

### 示例 1: s = "abc", t = "ahbgdc" -> true

用表格跟踪每一步 `for` 循环迭代。`s` 指针下方标注当前待匹配字符。

| 迭代 | *t | *s | *t == *s? | 动作 | s 指向 |
|------|----|----|-----------|------|--------|
| 1 | 'a' | 'a' | 是 | s++ | "bc" |
| 2 | 'h' | 'b' | 否 | 无 | "bc" |
| 3 | 'b' | 'b' | 是 | s++ | "c" |
| 4 | 'g' | 'c' | 否 | 无 | "c" |
| 5 | 'd' | 'c' | 否 | 无 | "c" |
| 6 | 'c' | 'c' | 是 | s++ | "" |

循环结束：`*t` 在下一步为 `'\0'`，退出 `for`。此时 `*s == '\0'`，故 `!*s` 为 `true`。返回 `true`。

### 示例 2: s = "axc", t = "ahbgdc" -> false

| 迭代 | *t | *s | *t == *s? | 动作 | s 指向 |
|------|----|----|-----------|------|--------|
| 1 | 'a' | 'a' | 是 | s++ | "xc" |
| 2 | 'h' | 'x' | 否 | 无 | "xc" |
| 3 | 'b' | 'x' | 否 | 无 | "xc" |
| 4 | 'g' | 'x' | 否 | 无 | "xc" |
| 5 | 'd' | 'x' | 否 | 无 | "xc" |
| 6 | 'c' | 'x' | 否 | 无 | "xc" |

循环结束：`*s == 'x'`，不是 `'\0'`，故 `!*s` 为 `false`。返回 `false`。`t` 中没有 `'x'`，所以 `s` 在第一个字符之后就再也推进不了了。

## 容易踩的坑

**(a) s 只在匹配时前进。** `t` 每次迭代都前进（由 `for` 的 `t++` 保证），但 `s` 只有在 `*t == *s` 成立时才前进。如果误把 `s++` 写在 `if` 外面，`s` 就会和 `t` 同步前进，变成判断"相等字符串"而非"子序列"。

**(b) `!*s` 的返回值含义。** `*s` 取的是 `s` 当前指向的字符。如果 `s` 的所有字符都已匹配完毕，`s` 会指向末尾的 `'\0'`，此时 `*s == 0`，`!*s` 为 `true`。如果还有未匹配的字符，`*s` 非零，`!*s` 为 `false`。这一行同时处理了"全部匹配"和"未全部匹配"两种情况，无需额外的 `if`。

**(c) 空 s 自然处理。** 当 `s` 为空字符串时，`*s` 一开始就是 `'\0'`。`for` 循环内的 `if (*t == *s)` 永远不会让 `*t`（可打印字符）等于 `'\0'`，所以 `s` 保持不动。循环结束后 `!*s` 仍为 `true`——空串是任何字符串的子序列。不需要任何特殊判断。

**(d) 无需显式"未找到"检查。** 原始版本需要在内层循环结束后检查 `found` 标志来决定是否提前返回 `false`。优化版把这个逻辑推迟到最后一步：`return !*s`。如果某个字符在 `t` 中找不到，`s` 就不会推进到末尾，最终 `!*s` 自然为 `false`。判断逻辑从"每步检查失败"变成了"最后检查成功"，更简洁也更不容易出错。

## 复杂度分析

| 指标 | 值 | 说明 |
|------|----|------|
| 时间复杂度 | O(\|t\|) | `for` 循环恰好遍历 `t` 一遍，每步 O(1) 操作。 |
| 空间复杂度 | O(1) | 只使用传入的两个指针，无额外分配。 |

## 与原始解法对比

| | 原始版本 | 优化版本 |
|--|---------|---------|
| 算法 | 嵌套双指针 + found 标志 | 单 for 循环扫描 |
| 时间 | O(\|s\| + \|t\|) | O(\|t\|) |
| 空间 | O(1) | O(1) |
| 循环结构 | 嵌套 while/while | 单个 for |
| 代码行数 | 17（函数体） | 4（函数体） |
| 额外变量 | p1, p2, found | 无（直接修改 s, t） |

两者在渐近意义上都是线性的，但优化版的常数因子更小：少一层循环调度，少一个分支判断，少三个局部变量。更重要的是，代码的**意图**变得一目了然——"扫一遍 `t`，顺便把 `s` 划掉"。

---

## 总结

优化版的核心洞察是：判断子序列本质上只需要扫描 `t` 一遍。原始版本以 `s` 为主视角，对每个字符在 `t` 中搜索，需要嵌套循环和状态标志来协调两个指针的推进。优化版切换到以 `t` 为主视角——`t` 无条件前进，`s` 仅在匹配时前进——从而将整个逻辑压缩为一个 `for` 循环加一条 `return` 语句。没有多余的变量，没有多余的分支，没有提前返回，每一行都直接服务于算法的核心意图。这正是费曼所说的"用最少的零件把事情说清楚"。
