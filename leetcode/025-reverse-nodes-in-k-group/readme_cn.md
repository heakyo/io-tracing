## 题目描述

给定一个单链表的头节点 `head` 和一个正整数 `k`，请你每 `k` 个节点一组进行翻转，并返回修改后的链表。如果链表末尾剩余的节点不足 `k` 个，则保持原有顺序不变。

**约束条件：**

- 链表节点数 `n` 满足 `1 <= n <= 5000`
- `1 <= k <= n`
- 节点值范围：`0 <= Node.val <= 1000`

**示例：**

```
输入: head = [1,2,3,4,5], k = 2
输出: [2,1,4,3,5]

输入: head = [1,2,3,4,5], k = 3
输出: [3,2,1,4,5]
```

用通俗的话讲：把链表切成若干段，每段 `k` 个节点，把每段内部的顺序翻转过来。最后一段如果不够 `k` 个，就不翻转，原样保留。

---

## 核心思路

这道题是 LeetCode 92（反转链表 II）的升级版——不再只反转一段，而是**反复地**对每一组 `k` 个节点做同样的"头插法"反转。

用大白话讲，算法分两步走：

### 第一步：先数清楚一共有多少个节点

```c
size = 0;
cur = head;
while (cur) {
    size++;
    cur = cur->next;
}
```

为什么要先遍历一遍？因为我们需要知道还剩多少个节点，以此判断能不能再凑满一组 `k` 个。如果剩余节点不足 `k` 个，就停下来，保持原序。

### 第二步：逐组头插法反转

```c
while (size >= k) {
    for (i = 0; i < k - 1; i++) {
        tmp = pre->next;
        pre->next = cur->next;
        cur->next = cur->next->next;
        pre->next->next = tmp;
    }
    size -= k;
    pre = cur;
    cur = cur->next;
}
```

每一组内部的反转手法和 LeetCode 92 完全一样：**反复把 `cur` 后面的节点摘下来，插到 `pre` 后面。** 每组翻转 `k - 1` 次之后，`cur` 自然下沉到该组的末尾，成为下一组的 `pre`，然后 `cur` 前进一步，开始下一组的反转。

想象一副扑克牌排成一行，每 `k` 张一组。对每一组，你从左到右依次把第 2 张、第 3 张......第 `k` 张抽出来，塞到这一组的最左边。抽完 `k - 1` 次，这一组的顺序就倒过来了。然后移动到下一组，重复同样的操作。

## 逐步推演

以 `head = [1,2,3,4,5], k = 2` 为例，完整走一遍。

### 初始化

```
dummy -> 1 -> 2 -> 3 -> 4 -> 5 -> NULL
pre = dummy, size = 5, cur = Node(1)
```

计数阶段遍历链表得到 `size = 5`，然后 `cur` 复位到 `head = Node(1)`。

### 第 1 组反转（size=5 >= k=2，循环 k-1=1 次）

#### i=0：摘出 Node(2)，插到 pre 后面

操作前：

```
pre    cur
 |      |
 v      v
dummy -> 1 -> 2 -> 3 -> 4 -> 5 -> NULL
```

四步指针操作：

```
tmp = pre->next              => tmp = Node(1)
pre->next = cur->next        => pre->next = Node(2)     -- 摘出 Node(2)
cur->next = cur->next->next  => cur->next = Node(3)     -- cur 跳过 Node(2)
pre->next->next = tmp        => Node(2)->next = Node(1)  -- Node(2) 指向原头部
```

操作后：

```
pre          cur
 |            |
 v            v
dummy -> 2 -> 1 -> 3 -> 4 -> 5 -> NULL
```

#### 本组收尾

```
size -= k   => size = 5 - 2 = 3
pre = cur   => pre = Node(1)
cur = cur->next => cur = Node(3)
```

```
          pre    cur
           |      |
           v      v
dummy -> 2 -> 1 -> 3 -> 4 -> 5 -> NULL
```

### 第 2 组反转（size=3 >= k=2，循环 k-1=1 次）

#### i=0：摘出 Node(4)，插到 pre 后面

操作前：

```
          pre    cur
           |      |
           v      v
dummy -> 2 -> 1 -> 3 -> 4 -> 5 -> NULL
```

四步指针操作：

```
tmp = pre->next              => tmp = Node(3)
pre->next = cur->next        => pre->next = Node(4)     -- 摘出 Node(4)
cur->next = cur->next->next  => cur->next = Node(5)     -- cur 跳过 Node(4)
pre->next->next = tmp        => Node(4)->next = Node(3)  -- Node(4) 指向原头部
```

操作后：

```
          pre               cur
           |                 |
           v                 v
dummy -> 2 -> 1 -> 4 -> 3 -> 5 -> NULL
```

#### 本组收尾

```
size -= k   => size = 3 - 2 = 1
pre = cur   => pre = Node(3)
cur = cur->next => cur = Node(5)
```

```
                    pre    cur
                     |      |
                     v      v
dummy -> 2 -> 1 -> 4 -> 3 -> 5 -> NULL
```

### 循环终止（size=1 < k=2）

剩余 1 个节点不足 `k=2`，保持原序，退出循环。

### 最终结果

```
返回 dummy->next: 2 -> 1 -> 4 -> 3 -> 5
```

### 完整变化总结

| 阶段 | 操作 | 链表状态 |
|------|------|---------|
| 初始 | -- | 1 -> 2 -> 3 -> 4 -> 5 |
| 第 1 组 (i=0) | 摘出 Node(2) | **2 -> 1** -> 3 -> 4 -> 5 |
| 第 2 组 (i=0) | 摘出 Node(4) | 2 -> 1 -> **4 -> 3** -> 5 |

## 易错点分析

### (a) 为什么需要 dummy 节点？

当 `k` 等于链表长度时，整条链表都要反转，第一个节点不再是头节点。如果没有 `dummy`，你就需要单独处理头节点变化的情况。`dummy` 提供了一个**稳定的锚点**——不管反转多少组，`dummy->next` 始终指向最新的链表头节点。

### (b) 为什么要先计数？

一种常见的替代方案是每次反转前先向前探 `k` 步，看看够不够。但代码中的做法更简洁：先一次性数清总量，然后每反转一组就 `size -= k`，用一个简单的整数比较来判断是否继续。两种方案的时间复杂度都是 O(n)，但计数法的实现更直观。

### (c) cur 为什么不移动？

这是头插法最反直觉的地方。在每组的反转循环内部，`cur` 始终指向**该组原来的第一个节点**。随着后面的节点不断被摘出并插到 `pre` 后面，`cur` 自然而然地"下沉"到该组的末尾。

反转结束后，`cur` 恰好在该组最后一个位置，正好可以作为下一组的 `pre`。

### (d) 四行指针操作的顺序不能乱

```c
tmp = pre->next;             // 1. 先保存当前反转段头部
pre->next = cur->next;       // 2. 摘出 cur 后面的节点
cur->next = cur->next->next; // 3. cur 跳过被摘出的节点
pre->next->next = tmp;       // 4. 被摘出的节点指向原头部
```

第 1 行必须最先执行，因为第 2 行会覆盖 `pre->next`。如果不先保存，原来的反转段头部就丢了，链表就断了。同理，第 3 行依赖第 2 行执行后 `cur->next` 仍然指向被摘出的节点这一事实，从而正确地跳过它。

### (e) 内存泄漏

```c
dummy = malloc(sizeof(*dummy));
// ...
return dummy->next;  // dummy 本身从未 free
```

`dummy` 是堆上分配的内存，但函数返回时只返回了 `dummy->next`，`dummy` 自身永远不会被释放。LeetCode 评测系统不检查内存泄漏，所以能通过，但在生产代码中应当修复：

```c
struct ListNode *result = dummy->next;
free(dummy);
return result;
```

## 复杂度分析

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 时间 | **O(n)** | 计数阶段遍历整个链表一次（n 步）。反转阶段，每个节点最多被操作一次（每组 k 个节点做 k-1 次指针操作），总计不超过 n 次。两个阶段加起来是 O(2n) = O(n) |
| 空间 | **O(1)** | 只使用了 `dummy`、`pre`、`cur`、`tmp` 四个指针和 `size`、`i` 两个整数变量。`dummy` 的 `malloc` 是 O(1) 常数空间（虽然存在泄漏） |

## 总结

K 个一组翻转链表的核心思想可以浓缩为一句话：

> **先数总量，然后逐组使用头插法反转，每组内 `cur` 不动，反复把 `cur` 后面的节点摘下来插到 `pre` 后面，做 `k-1` 次即可翻转一组。**

这个算法的巧妙之处在于：它把一个看似复杂的"多组反转"问题，分解成了重复执行"单组反转"的简单操作。每组反转结束后，`cur` 恰好下沉到该组末尾，天然成为下一组的 `pre`，无需额外的定位操作。整个过程只需要遍历链表两次（计数一次、反转一次），时间 O(n)、空间 O(1)，实现简洁高效。
