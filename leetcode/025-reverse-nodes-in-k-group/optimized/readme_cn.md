## 题目描述

给你链表的头节点 `head`，每 `k` 个节点一组进行翻转，请你返回修改后的链表。

`k` 是一个正整数，它的值小于或等于链表的长度。如果节点总数不是 `k` 的整数倍，
那么请将最后剩余的节点保持原有顺序。

不能只改变节点内部的值，需要进行实际的节点交换。

**示例：**

```
输入：head = [1,2,3,4,5], k = 2
输出：[2,1,4,3,5]

输入：head = [1,2,3,4,5], k = 3
输出：[3,2,1,4,5]
```

## 核心思路

算法分两步走：

1. **计数**——先遍历一遍链表，统计总节点数 `size`。
2. **分组翻转**——只要 `size >= k`，就用**头插法**翻转当前这一组 k 个节点，
   然后把指针推进到下一组。

头插法的关键：每次从当前组的"尾巴"后面抽出一个节点，插到"组头"前面。
对一组 k 个节点执行 `k - 1` 次这样的操作，整组就翻转了。

为什么先计数？因为题目要求**不足 k 个的尾部不翻转**。先知道总数，就能在
`while` 循环里直接判断，避免翻转了不该翻转的部分再撤销。

优化版在算法逻辑上与原版完全相同，改进集中在**代码质量**上——消除内存泄漏、
使用更清晰的变量名、让每行代码只做一件事。

关键代码：

```c
struct ListNode dummy = { .val = 0, .next = head };
struct ListNode *prev, *tail, *move, *cur;
int size, i;

/* 计数 */
size = 0;
for (cur = head; cur; cur = cur->next)
    size++;

prev = &dummy;
tail = head;

while (size >= k) {
    for (i = 0; i < k - 1; i++) {
        move = tail->next;
        tail->next = move->next;
        move->next = prev->next;
        prev->next = move;
    }

    size -= k;
    prev = tail;
    tail = tail->next;
}

return dummy.next;
```

三个核心指针的含义：

- **`prev`**——当前组的"前驱节点"，始终站在组外，负责把被搬运的节点接到组头。
- **`tail`**——当前组的"原始首节点"。翻转过程中它不移动，但因为不断有节点
  被插到它前面，它自然"下沉"到组尾。
- **`move`**——每轮被从 `tail` 后面抽出、插到 `prev` 后面的那个节点。

## 逐步推演

以 `[1,2,3,4,5]`，`k = 2` 为例。

### 初始化

先遍历链表统计 `size = 5`，然后设置起始指针：

```
dummy -> 1 -> 2 -> 3 -> 4 -> 5 -> NULL
  ^      ^
 prev   tail
```

### 第一组（size=5, 5 >= 2, 进入循环）

内层 `for` 循环执行 `k - 1 = 1` 次（`i = 0`）：

**抓取 move：**

```
move = tail->next            // move 指向节点 2

dummy -> 1 -> 2 -> 3 -> 4 -> 5 -> NULL
  ^      ^    ^
 prev  tail  move
```

**四步指针操作：**

```
tail->next = move->next      // 节点 1 跳过节点 2，指向节点 3
move->next = prev->next      // 节点 2 指向 prev 后面的节点（即节点 1）
prev->next = move            // prev 指向节点 2（节点 2 成为组头）
```

**结果：**

```
dummy -> 2 -> 1 -> 3 -> 4 -> 5 -> NULL
         ^    ^
  (prev      tail
   ->next)
```

> 节点 2 被"搬"到了组最前面，`tail`（节点 1）下沉到了组尾。

**推进指针到下一组：**

```
size -= k    =>  size = 3
prev = tail  =>  prev 指向节点 1
tail = tail->next  =>  tail 指向节点 3

dummy -> 2 -> 1 -> 3 -> 4 -> 5 -> NULL
              ^    ^
            prev  tail
```

### 第二组（size=3, 3 >= 2, 继续循环）

内层 `for` 循环执行 1 次（`i = 0`）：

**抓取 move：**

```
move = tail->next            // move 指向节点 4

dummy -> 2 -> 1 -> 3 -> 4 -> 5 -> NULL
              ^    ^    ^
            prev  tail move
```

**四步指针操作：**

```
tail->next = move->next      // 节点 3 跳过节点 4，指向节点 5
move->next = prev->next      // 节点 4 指向 prev 后面的节点（即节点 3）
prev->next = move            // prev 指向节点 4（节点 4 成为组头）
```

**结果：**

```
dummy -> 2 -> 1 -> 4 -> 3 -> 5 -> NULL
              ^         ^
            prev       tail
```

**推进指针到下一组：**

```
size -= k    =>  size = 1
prev = tail  =>  prev 指向节点 3
tail = tail->next  =>  tail 指向节点 5

dummy -> 2 -> 1 -> 4 -> 3 -> 5 -> NULL
                        ^    ^
                      prev  tail
```

### 退出循环（size=1, 1 >= 2 不成立）

剩余 1 个节点不足一组，保持原序。

### 最终结果

返回 `dummy.next`，即 `[2, 1, 4, 3, 5]`。

## 易错点分析

### (a) 栈 dummy vs malloc dummy

原版代码：

```c
dummy = malloc(sizeof(*dummy));
dummy->next = head;
/* ... */
return dummy->next;    /* dummy 没有 free，内存泄漏！ */
```

优化版代码：

```c
struct ListNode dummy = { .val = 0, .next = head };
/* ... */
return dummy.next;     /* dummy 在栈上，函数返回自动销毁 */
```

为什么栈上 dummy 可行？因为我们返回的是 `dummy.next`——一个指向堆上真实节点的
指针，而不是 `&dummy` 本身。函数返回后 `dummy` 虽然销毁了，但返回值已经拷贝
出去，完全安全。

### (b) tail 不移动——它在"下沉"

`tail` 在一组的翻转过程中始终指向**同一个节点**。它"看起来"在下沉，是因为
不断有新节点被插入到它前面。只有在一组翻转结束后，`tail` 才被显式推进到
下一组的首节点。

```
开始：prev -> [1] -> 2 -> 3 -> ...
                ^
              tail

结束：prev -> 2 -> [1] -> 3 -> ...
                    ^
                  tail（同一个节点，值还是 1）
```

如果你错误地在内层循环中移动了 `tail`（比如 `tail = tail->next`），就会从
错误的位置抽取节点，导致链表断裂。

### (c) 四行指针操作必须严格按序执行

```c
move = tail->next;          /* 1. 抓住要搬的节点 */
tail->next = move->next;    /* 2. tail 跳过 move */
move->next = prev->next;    /* 3. move 指向当前组头 */
prev->next = move;          /* 4. prev 指向 move（新的组头） */
```

如果打乱顺序——比如先执行第 4 步再执行第 3 步——`prev->next` 已经被改写，
第 3 步中 `move->next = prev->next` 就会指向错误的节点。

**口诀：抓、跳、接、插。**

### (d) 计数循环不能省

没有提前计数就无法判断剩余节点是否够一组。如果盲目翻转，最后不足 k 个的
尾部也会被翻转，违反题意。

### (e) while 条件是 `size >= k`，不是 `size > k`

`size >= k` 确保恰好剩 k 个节点时仍然翻转。如果写成 `size > k`，最后
恰好一组 k 个节点会被跳过。

## 与原版对比

| 对比维度 | 原版 | 优化版 |
|----------|------|--------|
| dummy 节点 | `malloc` 分配在堆上，函数返回时未 `free`，**内存泄漏** | 栈上声明 `struct ListNode dummy = {0}`，函数返回自动销毁，**零泄漏** |
| 变量命名 | `pre`（前驱）、`cur`（实际是 tail）、`tmp`（实际是被搬运的节点） | `prev`（前驱）、`tail`（组首 -> 组尾）、`move`（被搬运的节点） |
| 命名歧义 | `cur` 暗示"当前正在移动的指针"，但它在组内翻转时**并不移动**，容易误导 | `tail` 准确描述了它的行为——翻转后变为组尾 |
| 计数循环 | `while` 循环 + 手动 `size++` + 手动 `cur = cur->next`，3 行 | `for` 循环一行搞定，更紧凑 |
| 翻转循环体 | 涉及 `cur->next->next` 链式访问，不易一眼看清 | 每行只做一件事——抓、跳、接、插，逻辑更直白 |
| 算法本质 | 头插法 | 头插法（完全相同） |
| 时间复杂度 | O(n) | O(n) |
| 空间复杂度 | O(1)（但有堆泄漏） | **真正 O(1)**——所有辅助变量都在栈上 |

原版翻转循环体：

```c
tmp = pre->next;
pre->next = cur->next;
cur->next = cur->next->next;
pre->next->next = tmp;
```

优化版翻转循环体：

```c
move = tail->next;
tail->next = move->next;
move->next = prev->next;
prev->next = move;
```

两段代码做的事情完全一样，但优化版先把 `tail->next` 保存到 `move` 变量里，
避免了 `cur->next->next` 这种链式解引用，每行只修改一个指针，读起来更清晰。

## 复杂度分析

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 时间 | **O(n)** | 计数循环遍历一遍 n 个节点；翻转阶段每个节点最多被操作一次（每个节点在内层循环中要么作为 `move` 被搬运一次，要么作为 `tail` 被跳过一次）。总操作次数为 n，因此 O(n) |
| 空间 | **O(1)** | 只用了 `dummy`、`prev`、`tail`、`move`、`cur`、`size`、`i` 这几个局部变量。`dummy` 在栈上，**真正的 O(1)**——不像原版用 `malloc` 分配了一个堆节点 |

## 函数参考

以下是 `optimized/main.c` 中的所有函数：

### `reverseKGroup`

```c
struct ListNode* reverseKGroup(struct ListNode* head, int k);
```

- **用途**：核心算法。每 k 个节点一组，用头插法原地翻转链表。不足 k 个的尾部
  保持原序。
- **参数**：
  - `head`——链表头节点指针
  - `k`——每组的节点数
- **返回值**：翻转后的链表头节点指针

### `insert_list`

```c
struct ListNode *insert_list(int *nums, int size);
```

- **用途**：从整型数组构建单链表。从尾到头依次 `malloc` 并头插。
- **参数**：
  - `nums`——整型数组
  - `size`——数组长度
- **返回值**：链表头节点指针

### `show_list`

```c
void show_list(char *type, struct ListNode* head);
```

- **用途**：遍历并打印链表内容，格式为 `type: [ val1 val2 ... ]`。
- **参数**：
  - `type`——标签字符串（如 `"Input "` / `"Output"`）
  - `head`——链表头节点指针
- **返回值**：无

### `free_list`

```c
static void free_list(struct ListNode *head);
```

- **用途**：遍历链表，逐节点 `free`，释放所有堆内存。
- **参数**：
  - `head`——链表头节点指针
- **返回值**：无

### `check_list`

```c
static int check_list(struct ListNode *head, int *expected, int len);
```

- **用途**：将链表与期望数组逐元素比较，长度也必须匹配。
- **参数**：
  - `head`——链表头节点指针
  - `expected`——期望值数组
  - `len`——期望值数组长度
- **返回值**：1 表示匹配通过，0 表示不匹配

### `run_test`

```c
static void run_test(const char *name, int *nums, int len,
                     int k, int *expected, int expected_len);
```

- **用途**：执行一个测试用例——构建链表、调用 `reverseKGroup`、打印结果、
  用 `assert` 校验、最后释放内存。
- **参数**：
  - `name`——测试用例名称
  - `nums`——输入数组
  - `len`——输入数组长度
  - `k`——每组节点数
  - `expected`——期望输出数组
  - `expected_len`——期望输出数组长度
- **返回值**：无

## 总结

优化版和原版解决的是**同一个问题**，用的是**同一个算法**（先计数、再分组头插），
时间复杂度都是 O(n)。改进集中在**代码质量**上：栈上 `dummy` 节点消除了原版的
内存泄漏；`prev`/`tail`/`move` 三个变量名精确对应各自角色——前驱、下沉节点、
被搬运节点；翻转循环体每行只做一件事，没有 `cur->next->next` 这种链式访问。
最终达到的效果是：代码行为不变，但可读性显著提高，维护成本更低。
