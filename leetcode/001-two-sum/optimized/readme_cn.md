## 为什么要优化？

原始暴力解法用两层嵌套循环尝试所有配对，时间复杂度为 O(n^2)。当数组很大时（比如 10,000 个元素），需要多达 5000 万次比较。我们可以用少量额外内存换取大幅提速。

---

## 核心思路

想象同样那个房间，每个人手里举着数字卡片。这次不再逐对比较，而是在门口放一块**公告板**（哈希表）。每个人进门时：

1. 算出自己需要配对的数字：`补数 = target - 我的数字`。
2. 看一眼公告板 —— 补数已经贴上去了吗？
3. 如果有，找到配对了！如果没有，把自己的数字和位置贴到公告板上，继续走。

这样每个人只需看一次公告板，走一遍房间就搞定了。

在代码中，"公告板"就是哈希表，存储 `数字 -> 下标` 的映射。对于每个元素，计算 `complement = target - nums[i]`，在哈希表中查找，找到就返回结果，找不到就把当前元素插入哈希表。

## 逐步演练

`nums = [3, 2, 3]`，`target = 6`：

```
哈希表: {}

i=0: nums[0]=3, 补数=6-3=3
     查找 3 -> 没找到
     插入 {3: 0}

i=1: nums[1]=2, 补数=6-2=4
     查找 4 -> 没找到
     插入 {3: 0, 2: 1}

i=2: nums[2]=3, 补数=6-3=3
     查找 3 -> 找到了! 下标=0
     返回 [0, 2]
```

另一个例子：`nums = [2, 7, 11, 15]`，`target = 9`：

```
哈希表: {}

i=0: nums[0]=2, 补数=9-2=7
     查找 7 -> 没找到
     插入 {2: 0}

i=1: nums[1]=7, 补数=9-7=2
     查找 2 -> 找到了! 下标=0
     返回 [0, 1]
```

## 容易踩的坑

1. **先查找，后插入。** 如果先把 `nums[i]` 插入哈希表再查找补数，可能会把元素和自己配对。比如 `nums = [3, 3]`，`target = 6`：如果先插入，`i=0` 就会在表中找到自己。先查找后插入，第一个 `3` 找不到匹配，第二个 `3` 才会正确找到第一个。

2. **哈希冲突。** C 语言没有内置哈希表，所以我们用开放寻址法（线性探测）自己实现。哈希表大小要足够大以减少冲突，这里用了 4096 个槽位，对 LeetCode 的典型输入绰绰有余。

3. **负数键值。** 哈希函数必须正确处理负整数。我们在计算前将 `int` 转为 `unsigned int`，避免负数取模的未定义行为。

4. **结果顺序。** 哈希表方法天然返回 `[较小下标, 较大下标]`，因为我们总是先找到之前插入的那个元素。

## 复杂度分析

| 指标 | 值 | 原因 |
|------|-----|------|
| **时间** | O(n) | 遍历数组一次；每次哈希表查找/插入均摊 O(1) |
| **空间** | O(n) | 哈希表最多存储 n 个元素 |

---

## 与原始解法对比

| | 原始（暴力法） | 优化（哈希表） |
|--|----------------|----------------|
| **时间** | O(n^2) | O(n) |
| **空间** | O(1) | O(n) |
| **思路** | 两层循环尝试所有配对 | 一次遍历 + 哈希表查找 |
| **代码复杂度** | 非常简单 | 需要在 C 中手动实现哈希表 |

这是经典的权衡：**用空间换时间**。对于这道题，哈希表方法是公认的最优解。

---

## 总结

优化解法用单次遍历加哈希表替代了 O(n^2) 的暴力搜索。对于每个元素，计算 `target - nums[i]`，在哈希表中查找：找到了就返回答案，没找到就把当前元素存入哈希表。时间从 O(n^2) 降到 O(n)，代价是 O(n) 的额外空间 —— 经典的时空权衡。在 C 语言中，我们用开放寻址法和简单的整数哈希函数手动实现了哈希表。
