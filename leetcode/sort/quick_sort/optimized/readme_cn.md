## 为什么要优化？

原始快速排序有三个影响性能的问题：

1. **左递归缺陷**：`quick_sort(array, 0, pivot - 1)` 在每次递归时都从索引 0 开始重新排序整个前缀，而不是只排序从 `low` 开始的当前子区间。这使得平均 O(n log n) 在实际中退化为 O(n^2)。

2. **固定基准选择**：总是选最后一个元素作为基准，在已排序或逆序输入上退化为 O(n^2)——而这恰恰是现实中最常见的模式之一。

3. **无小数组优化**：对于很小的子数组，递归开销占主导地位，用更简单的算法会更快。

---

## 核心思路

把原始方法想象成每次都让队尾的人当"分界线"。如果队伍已经排好序，这个人总是最大的（或最小的），队伍几乎分不开——效率极低。

优化版做了三个改进：

1. **三数取中基准选择**：不再盲目选最后一个元素，而是看队头、队中、队尾三个元素，选中间值。这避免了已排序输入上的最坏情况，通常能得到更均衡的分区。

2. **小数组用插入排序**：当子数组不超过 10 个元素时，停止递归，改用插入排序。插入排序开销更小，对于很小的数组实际上比快速排序更快。

3. **尾调用消除**：不用两次递归调用，改用 while 循环：对较小的一半递归，对较大的一半迭代。这保证递归栈深度不超过 O(log n)，即使在最坏情况下也是如此。

当然，左递归也正确地使用了 `low` 而不是 `0`。

## 函数说明

| 函数 | 签名 | 作用 |
|------|------|------|
| `swap` | `void swap(int *a, int *b)` | 通过临时变量交换两个整数。 |
| `insertion_sort` | `void insertion_sort(int *array, int low, int high)` | 用插入排序对 `array[low..high]` 排序。用于不超过 10 个元素的子数组，以避免递归开销。 |
| `median_of_three` | `int median_of_three(int *array, int low, int high)` | 对 `array[low]`、`array[mid]`、`array[high]` 三个元素排序，然后将中位数移到 `array[high-1]` 作为基准。返回基准值。确保 `array[low] <= pivot <= array[high]`，为分区扫描提供哨兵。 |
| `partition` | `int partition(int *array, int low, int high)` | 围绕三数取中基准对 `array[low..high]` 分区。使用 Hoare 风格双指针扫描：`i` 向右跳过 < pivot 的元素，`j` 向左跳过 > pivot 的元素，两者都停下时交换。返回基准的最终索引。 |
| `quick_sort` | `void quick_sort(int *array, int low, int high)` | 主排序函数。使用 while 循环（尾调用消除），总是对较小分区递归、对较大分区迭代。子数组大小 <= 10 时退化为插入排序。 |
| `print_array` | `void print_array(const char *label, int *array, int len)` | 带标签打印数组，如 `Input:  [1 7 3 8 4]`。 |
| `check_sorted` | `int check_sorted(int *array, int len)` | 验证数组是否按非递减顺序排列。已排序返回 1，否则返回 0。 |
| `run_test` | `void run_test(const char *name, int *src, int len)` | 执行单个测试：复制输入、排序、用 `check_sorted` 和 `assert` 验证、打印 PASS。 |
| `main` | `int main(int argc, char *argv[])` | 程序入口。运行全部 10 个测试用例（与原始版本完全相同），成功后打印 "All tests passed!"。 |

## 逐步演练

以 `[3, 1, 4, 1, 5, 9, 2, 6, 5, 3]`（10 个元素）为例，刚好触发插入排序阈值：

```
quick_sort(arr, 0, 9): 大小=10 <= INSERTION_CUTOFF
  -> insertion_sort(arr, 0, 9)

插入排序过程:
  i=1: key=1, 将 3 右移 -> [1, 3, 4, 1, 5, 9, 2, 6, 5, 3]
  i=2: key=4, 已在正确位置 -> [1, 3, 4, 1, 5, 9, 2, 6, 5, 3]
  i=3: key=1, 将 4,3 右移 -> [1, 1, 3, 4, 5, 9, 2, 6, 5, 3]
  i=4: key=5, 已在正确位置 -> [1, 1, 3, 4, 5, 9, 2, 6, 5, 3]
  i=5: key=9, 已在正确位置 -> [1, 1, 3, 4, 5, 9, 2, 6, 5, 3]
  i=6: key=2, 将 9,5,4,3 右移 -> [1, 1, 2, 3, 4, 5, 9, 6, 5, 3]
  i=7: key=6, 将 9 右移 -> [1, 1, 2, 3, 4, 5, 6, 9, 5, 3]
  i=8: key=5, 将 9,6 右移 -> [1, 1, 2, 3, 4, 5, 5, 6, 9, 3]
  i=9: key=3, 将 9,6,5,5,4 右移 -> [1, 1, 2, 3, 3, 4, 5, 5, 6, 9]

结果: [1, 1, 2, 3, 3, 4, 5, 5, 6, 9]
```

再看一个 11 个元素的例子，触发分区逻辑。以 `[8, 3, 7, 1, 5, 9, 2, 6, 4, 10, 0]` 为例：

```
quick_sort(arr, 0, 10): 大小=11 > 10

三数取中: low=0(8), mid=5(9), high=10(0)
  排序: 0, 8, 9 -> arr[0]=0, arr[5]=8, arr[10]=9
  将 arr[5] 和 arr[9] 交换: pivot_val=8, 基准在索引 9
  数组: [0, 3, 7, 1, 5, 8, 2, 6, 4, 8, 9]
                                     ^ 基准在索引 9

partition(0, 10):
  i=0, j=9 (从 i+1=1 和 j-1=8 开始扫描)
  i 右扫: arr[1]=3<8, arr[2]=7<8, arr[3]=1<8, arr[4]=5<8,
          arr[5]=8 不<8, 停在 i=5
  j 左扫: arr[8]=4 不>8, 停在 j=8
  i(5) < j(8): 交换 arr[5] 和 arr[8] -> [..., 4, 2, 6, 8, ...]
  继续: i 扫: arr[6]=2<8, arr[7]=6<8, arr[8]=8 不<8, i=8
        j 扫: arr[7]=6 不>8, j=7
  i(8) >= j(7): 跳出
  交换 arr[8] 和 arr[9]（基准）: 基准现在在索引 8

左半部分 (0..7), 右半部分 (9..10)
右半部分更小 -> 递归(9,10), 设 high=7

  递归(9,10): 大小=2 <= 10 -> insertion_sort
  回到 while 循环, low=0, high=7: 大小=8 <= 10 -> insertion_sort
```

## 容易踩的坑

1. **三数取中需要至少 3 个元素。** 代码只在子数组超过 10 个元素时才调用 `partition`（进而调用 `median_of_three`），所以总是安全的。如果把 `INSERTION_CUTOFF` 降到 3 以下，三数取中逻辑就需要额外的边界检查。

2. **分区扫描中的哨兵元素。** 经过 `median_of_three` 后，`array[low] <= pivot <= array[high]`。这意味着左扫描（`while array[++i] < pivot`）最远到 `high-1` 就会停下，右扫描（`while array[--j] > pivot`）最远到 `low` 就会停下。不需要边界检查——排好序的端点充当了天然的哨兵。

3. **相同元素的处理。** 当很多元素等于基准值时，两个扫描指针都会停下并交换——这使得即使有大量重复元素，分区也保持均衡。与 Lomuto 分区（在全相同输入上退化为 O(n^2)）不同，这种 Hoare 风格的方法能优雅地处理相同元素。

4. **尾调用消除的正确性。** 总是对较小的一半递归、对较大的一半迭代，保证了 O(log n) 的最大栈深度，因为较小的一半最多有 n/2 个元素。如果没有这个优化，精心构造的输入可能导致 O(n) 的栈深度和栈溢出。

## 复杂度分析

| 指标 | 值 | 原因 |
|------|-----|------|
| **时间（最好/平均）** | O(n log n) | 三数取中在大多数输入上给出均衡的分区 |
| **时间（最坏）** | O(n^2) | 对精心构造的恶意输入仍可能出现，但三数取中使其在自然数据上极不可能发生 |
| **空间（栈）** | O(log n) 保证 | 尾调用消除确保总是对较小的一半递归 |

## 与原始解法对比

| | 原始版本 | 优化版本 |
|--|---------|---------|
| **左递归** | `quick_sort(array, 0, pivot-1)` — 每次都从索引 0 重新排序 | `quick_sort(array, low, pivot-1)` — 只排序当前子区间 |
| **基准选择** | 总是选最后一个元素 — 已排序输入上 O(n^2) | 三数取中 — 避免已排序/逆序输入上的最坏情况 |
| **小数组** | 一直递归到 1 个元素 | 不超过 10 个元素时用插入排序 — 开销更小 |
| **栈深度** | 最坏 O(n) | 通过尾调用消除保证 O(log n) |
| **分区方式** | 基准游走（基准通过交换移动到位） | Hoare 风格双指针 + 哨兵元素 |

---

## 总结

优化版快速排序修复了原始版本的 `0` vs `low` 缺陷，并增加了三项标准优化：三数取中基准选择以避免在已排序输入上退化到 O(n^2)；小数组用插入排序以减少递归开销；尾调用消除以保证 O(log n) 栈深度。Hoare 风格的分区从两端扫描，利用三数取中后的端点作为天然哨兵，无需边界检查。这些改动组合在一起，使算法对常见的最坏情况模式具有鲁棒性，同时保持 O(n log n) 的平均性能。
