## 题目描述

给定一个未排序的整数数组，使用快速排序算法将其按升序排列。

示例：

```
输入:  [1, 7, 3, 8, 4]
输出:  [1, 3, 4, 7, 8]
```

---

## 核心思路

想象你手里有一叠乱序的数字卡片，要把它们排好序。你随便抽出一张卡片——这就是**基准值（pivot）**——然后把剩下的卡片分成两堆：比基准小的放左边，比基准大的放右边。接着对左右两堆重复这个过程，直到每堆只剩一张卡片。把所有卡片摞在一起，就排好序了。

这段代码选择**最后一个元素**作为基准值，用双指针法进行分区。两个指针 `i` 和 `j` 分别从两端出发，`i` 向右扫描寻找比基准大的元素，`j` 向左扫描寻找比基准小的元素。找到后就和基准交换位置。基准在数组中不断"移动"，直到 `i` 和 `j` 相遇——此时基准就到了它最终的正确位置。

分区完成后，函数分别对左右两半递归处理。

## 逐步演练

以 `[1, 7, 3, 8, 4]` 为例：

```
初始状态: [1, 7, 3, 8, 4]   pivot_idx=4 (值为4), i=0, j=4

i 向右扫描: array[0]=1 <= 4, i++ -> i=1
            array[1]=7 > 4, 停止
  i(1) < pivot(4): 交换 array[1] 和 array[4] -> [1, 4, 3, 8, 7]  pivot_idx=1

j 向左扫描: array[1]=4 <= array[4]=7, j-- -> j=3
            array[1]=4 <= array[3]=8, j-- -> j=2
            array[1]=4 <= array[2]=3? 不满足 (4 > 3), 停止
  pivot(1) < j(2): 交换 array[1] 和 array[2] -> [1, 3, 4, 8, 7]  pivot_idx=2

检查: i=1, j=2, i < j? 是，继续外层循环。

i 向右扫描: i=1, array[1]=3 <= array[2]=4, i++ -> i=2
  i(2) == pivot(2): 停止，i < pivot 不成立，不交换

j 向左扫描: j=2, pivot(2) < j(2) 不成立，不扫描，不交换

检查: i=2, j=2, i < j? 否，退出循环。pivot_idx=2

递归左半部分: quick_sort([1, 3, 4, 8, 7], 0, 1)
递归右半部分: quick_sort([1, 3, 4, 8, 7], 3, 4)
```

左半部分 `[1, 3]`：基准值=3，位于索引1。1 <= 3，指针相遇，已经有序。

右半部分 `[8, 7]`：基准值=7，位于索引4。8 > 7，交换 -> `[7, 8]`。排好了。

最终结果：`[1, 3, 4, 7, 8]`

## 容易踩的坑

1. **基准值在数组中"游走"。** 与教科书上的 Lomuto 或 Hoare 分区不同，这段代码把基准值本身和其他元素交换，因此 `pivot_idx` 在循环中会变化。你必须追踪基准值的**当前位置**——不仅仅是它的值——因为两个指针都是和 `array[pivot_idx]` 比较的。

2. **左递归从 0 开始而非 `low`。** 第 44 行调用 `quick_sort(array, 0, pivot - 1)` 而不是 `quick_sort(array, low, pivot - 1)`。这不会破坏正确性——对已排序的前缀重新排序不会改变结果——但会做大量冗余工作。每次递归调用时，左分区都从数组开头开始处理，而不是当前子区间。这使得本应 O(n log n) 的平均复杂度在实际中退化为 O(n^2)。

3. **基准值总是选最后一个元素。** 选择 `high` 作为基准值，在已排序或逆序输入上会退化为 O(n^2)，因为每次分区都把所有元素放到同一侧。

4. **全部相同的元素。** 当所有元素都相等时，每次扫描都会遍历整个子数组才让指针相遇。基准值最终在一端，每层递归只减少一个元素——O(n^2) 的行为。

---

## 复杂度分析

| 指标 | 值 | 原因 |
|------|-----|------|
| **时间（最好/平均）** | O(n log n) | 每次分区大致将数组一分为二；log n 层递归，每层 O(n) 工作量 |
| **时间（最坏）** | O(n^2) | 已排序输入或全相同元素导致极度不平衡的分区 |
| **空间** | O(log n) 平均, O(n) 最坏 | 递归栈深度与分区平衡度相关 |

注意：左递归中 `0` 和 `low` 的问题（见"容易踩的坑"第 2 点）使得实际性能比标准快速排序更差，即使在随机输入上也是如此。

---

## 总结

这个快速排序实现使用了一种"基准游走"式分区方案：基准从最后一个元素开始，通过与两端指针汇聚过程中的交换，物理地移动到最终位置。基准就位后，函数对左右两半递归排序。算法能正确排序数组，但有一个性能缺陷——左递归总是从索引 0 而非 `low` 开始，导致冗余的重复排序。加上固定选最后一个元素作为基准，在已排序或全相同元素的输入上会退化为 O(n^2)。优化版本在 `optimized/` 目录中修复了这些问题。
