## 为什么要优化？

原始版本每次遇到不重复的元素时，调用 `swap()` 来交换 `nums[k]` 和 `nums[i]`。一次 swap 需要 3 次赋值操作（写入临时变量、写入 a、写入 b）。但仔细想一想：位置 `k` 之后的元素全是"垃圾数据"，LeetCode 根本不关心那些位置上存着什么值。既然 `nums[k]` 的旧值不需要保留，我们就没必要做交换，直接一次赋值 `nums[k] = nums[i]` 就够了——从 3 次赋值缩减到 1 次。

除此之外，优化版还带来了几个改进：

- 消除了 `swap()` 辅助函数，减少了函数调用开销和代码量。
- 增加了空数组保护：当 `numsSize == 0` 时提前返回 0，而原始版本会返回 1（这是一个 bug）。
- 使用 `nums[k++] = nums[i]` 把赋值和指针推进合并成一条语句，代码更紧凑。

一句话：我们用更少的操作完成了同样的事。

## 核心思路

和原始版本一样，优化版依然使用**双指针**策略：

- 慢指针 `k`：指向下一个要写入不重复元素的位置。
- 快指针 `i`：从左到右扫描整个数组。

每当 `nums[i] != nums[k-1]`（即发现了一个新的不重复值），就直接把 `nums[i]` 拷贝到 `nums[k]` 的位置，然后 `k` 向前推进一步。

唯一的区别是把 swap 替换成了直接赋值，并在函数开头对空数组做了提前返回。

```c
int removeDuplicates(int *nums, int numsSize)
{
	int i, k;

	if (numsSize == 0)
		return 0;

	for (i = 1, k = 1; i < numsSize; i++) {
		if (nums[i] != nums[k - 1])
			nums[k++] = nums[i];
	}

	return k;
}
```

## 逐步演练

### 例 1：`[1, 1, 2]`

初始状态：`k = 1`，从 `i = 1` 开始扫描。

| 步骤 | i | nums[i] | nums[k-1] | 相等？ | 操作 | k | 数组状态 |
|------|---|---------|-----------|--------|------|---|----------|
| 1 | 1 | 1 | 1 | 是 | 跳过 | 1 | `[1, 1, 2]` |
| 2 | 2 | 2 | 1 | 否 | `nums[1] = 2`; `k++` | 2 | `[1, 2, 2]` |

循环结束，返回 `k = 2`。前 2 个元素 `[1, 2]` 即为结果。

### 例 2：`[0, 0, 1, 1, 1, 2, 2, 3, 3, 4]`

初始状态：`k = 1`，从 `i = 1` 开始扫描。

| 步骤 | i | nums[i] | nums[k-1] | 相等？ | 操作 | k | 数组状态 |
|------|---|---------|-----------|--------|------|---|----------|
| 1 | 1 | 0 | 0 | 是 | 跳过 | 1 | `[0, 0, 1, 1, 1, 2, 2, 3, 3, 4]` |
| 2 | 2 | 1 | 0 | 否 | `nums[1] = 1`; `k++` | 2 | `[0, 1, 1, 1, 1, 2, 2, 3, 3, 4]` |
| 3 | 3 | 1 | 1 | 是 | 跳过 | 2 | `[0, 1, 1, 1, 1, 2, 2, 3, 3, 4]` |
| 4 | 4 | 1 | 1 | 是 | 跳过 | 2 | `[0, 1, 1, 1, 1, 2, 2, 3, 3, 4]` |
| 5 | 5 | 2 | 1 | 否 | `nums[2] = 2`; `k++` | 3 | `[0, 1, 2, 1, 1, 2, 2, 3, 3, 4]` |
| 6 | 6 | 2 | 2 | 是 | 跳过 | 3 | `[0, 1, 2, 1, 1, 2, 2, 3, 3, 4]` |
| 7 | 7 | 3 | 2 | 否 | `nums[3] = 3`; `k++` | 4 | `[0, 1, 2, 3, 1, 2, 2, 3, 3, 4]` |
| 8 | 8 | 3 | 3 | 是 | 跳过 | 4 | `[0, 1, 2, 3, 1, 2, 2, 3, 3, 4]` |
| 9 | 9 | 4 | 3 | 否 | `nums[4] = 4`; `k++` | 5 | `[0, 1, 2, 3, 4, 2, 2, 3, 3, 4]` |

循环结束，返回 `k = 5`。前 5 个元素 `[0, 1, 2, 3, 4]` 即为结果。位置 5 之后的值是垃圾数据，无需关心。

## 容易踩的坑

### (a) 赋值 vs swap —— 1 次写入代替 3 次

原始版本的 `swap(&nums[k], &nums[i])` 执行 3 次赋值：

```c
tmp = *a;    // 第 1 次
*a  = *b;    // 第 2 次
*b  = tmp;   // 第 3 次
```

但 `nums[k]` 的旧值并不需要保留（位置 k 及之后的数据对最终结果没有意义）。所以直接 `nums[k] = nums[i]` 只需要 1 次赋值，减少了 2/3 的写入操作。

### (b) `nums[k++] = nums[i]` 组合写入和推进

这条语句做了两件事：先把 `nums[i]` 写入 `nums[k]`，然后 `k` 自增。这是 C 语言中常见的惯用写法——后置 `++` 保证先使用 `k` 的当前值作为下标，再把 `k` 加 1。初学者容易把它误读成"先加 1 再赋值"，但后置自增的语义是：表达式的值是自增前的值。

### (c) 空数组保护

原始版本没有处理 `numsSize == 0` 的情况。当传入空数组时，循环不执行，函数返回 `k = 1`，这显然是错误的——空数组去重后应该返回 0。优化版在函数开头加了一个判断，提前返回 0，避免了这个 bug。

### (d) k == i 时自我拷贝无害

当数组中没有重复元素时，`k` 和 `i` 始终同步前进，此时 `nums[k] = nums[i]` 实际上是把元素赋值给自己（自我拷贝）。这在逻辑上完全无害，不会破坏数据。虽然可以加一个 `if (k != i)` 来跳过这种情况，但额外的分支判断带来的开销通常比一次冗余赋值更大，所以不加反而更好。

## 复杂度分析

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 时间 | O(n) | 快指针 `i` 从 1 遍历到 `numsSize - 1`，共 n-1 次比较 |
| 空间 | O(1) | 只使用了 `i` 和 `k` 两个额外变量，不依赖输入规模 |

## 与原始解法对比

| | 原始版本 | 优化版本 |
|--|---------|---------|
| 移动操作 | swap（3 次赋值） | 直接赋值（1 次赋值） |
| 额外函数 | swap() 辅助函数 | 无 |
| 空数组 | 返回 1（错误） | 返回 0（正确） |
| 代码行数 | 14（swap + 函数体） | 8（函数体） |
| 时间 | O(n) | O(n) |
| 空间 | O(1) | O(1) |

## 总结

优化版本的核心改动只有一个：把 swap 替换成直接赋值。这个改动之所以成立，是因为我们想清楚了一件事——位置 `k` 之后的元素是"不在乎"的垃圾数据，根本没必要保留它们的旧值。理解了这一点，swap 的第三次赋值（把旧值存回去）就变得多余了，于是 3 次赋值缩减为 1 次。同时，我们修复了原始版本在空数组上的 bug，去掉了 swap 辅助函数，用 `nums[k++] = nums[i]` 把写入和推进合二为一。最终，代码从 14 行缩减到 8 行，逻辑更清晰，性能更好，正确性也更完善——这就是"想清楚再写"带来的好处。
