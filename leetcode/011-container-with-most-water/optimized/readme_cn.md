## 为什么要优化？

原始版本的双指针解法，每次只把较矮一侧的指针往里移动 1 步。这已经是正确的了，但是
想一想：如果你刚才站在 `left` 这条高度为 `h` 的线上，往右挪了一步，发现新线等高甚至
更矮，有没有可能形成更大的容器？不可能。因为宽度减小了 1，而 `min` 高度没有增加，
面积只会更小或持平。所以这一步的面积计算是白做的。

优化版的想法很简单：既然等高或更矮的线不可能产生更大面积，那就别一步一步挪了。用一个
内层 `while` 循环，把较矮一侧所有高度 `<= h` 的线统统跳过，直接跳到下一条严格更高
的线再停下来。这样做有两个好处：

- 减少了外层循环的迭代次数（面积计算次数随之减少）。
- 不再需要 `MIN` 宏 -- 因为我们已经知道哪边更矮，直接用那边的高度就行。

## 核心思路

和原始版本一样，使用左右双指针从两端向中间收拢。不同之处在于：每次计算完面积之后，
不是简单地把较矮指针移动 1 步，而是用一个内层 `while` 跳过所有高度 `<= h` 的线。
只有当遇到一条严格更高的线时才停下来，重新进入外层循环做下一次面积计算。

为什么跳过是安全的？假设当前较矮侧高度为 `h`，宽度为 `w`。跳过的每一条线高度
`h' <= h`，而宽度 `w' < w`。面积 `h' * w' <= h * w'< h * w`，严格小于当前面积，
更不可能超过已经记录的最大面积。所以跳过不会漏掉答案。

## 逐步演练

输入: `[1, 8, 6, 2, 5, 4, 8, 3, 7]`

```
索引:   0  1  2  3  4  5  6  7  8
高度:   1  8  6  2  5  4  8  3  7
```

初始状态: `left=0, right=8, max_area=0`

---

**第 1 轮** (`left=0, right=8`)

- `height[0]=1 < height[8]=7`，走左分支。
- `h = height[0] = 1`。
- `area = 1 * (8 - 0) = 8 > 0`，更新 `max_area = 8`。
- 内层跳过: `height[0]=1 <= 1`，`left++` --> `left=1`。`height[1]=8 > 1`，停下。
- 结果: `left=1, right=8, max_area=8`。

---

**第 2 轮** (`left=1, right=8`)

- `height[1]=8 > height[8]=7`，走右分支。
- `h = height[8] = 7`。
- `area = 7 * (8 - 1) = 49 > 8`，更新 `max_area = 49`。
- 内层跳过: `height[8]=7 <= 7`，`right--` --> `right=7`。`height[7]=3 <= 7`，`right--` --> `right=6`。`height[6]=8 > 7`，停下。
- 结果: `left=1, right=6, max_area=49`。`right` 从 8 跳到 6，一次跳了 2 步。

---

**第 3 轮** (`left=1, right=6`)

- `height[1]=8 >= height[6]=8`，走右分支。
- `h = height[6] = 8`。
- `area = 8 * (6 - 1) = 40 < 49`，`max_area` 不变。
- 内层跳过: `height[6]=8 <= 8`，`right--` --> `right=5`。`height[5]=4 <= 8`，`right--` --> `right=4`。`height[4]=5 <= 8`，`right--` --> `right=3`。`height[3]=2 <= 8`，`right--` --> `right=2`。`height[2]=6 <= 8`，`right--` --> `right=1`。此时 `left=1, right=1`，`left < right` 为假，停下。
- 结果: `left=1, right=1`。`right` 一路从 6 跳到 1。

---

外层循环 `left < right` 为假，退出。返回 `max_area = 49`。

**只用了 3 次外层迭代，而原始版本需要 8 次！** 原始版本在第 2 轮之后会逐个移动
`right` 指针经过索引 7、6、5、4、3、2，每次都要计算一次面积。优化版直接跳过了
这些不可能产生更大面积的位置。

## 容易踩的坑

**(a) 内层跳过循环必须检查 `left < right`**

如果只写 `while (height[left] <= h)` 而不加 `left < right` 的守卫条件，当所有
剩余元素都 `<= h` 时，`left` 会越过 `right` 继续往后跑，造成越界访问。上面
第 3 轮的演练就展示了这种情况: `right` 一路减到和 `left` 相等才停下。

**(b) 用 `<=` 而非 `<` 跳过**

跳过条件是 `height[left] <= h`，不是 `height[left] < h`。等高的线也要跳过。
因为如果新线高度和 `h` 相同，宽度却更窄了，面积只会更小。只有严格更高的线才有
可能带来更大面积。如果错写成 `<`，遇到连续等高的线就会原地不动，陷入死循环。

**(c) 不需要 `MIN` 宏**

原始版本用 `MIN(height[left], height[right])` 来取两边的较小值。优化版通过
`if-else` 分支已经确定了哪边更矮，直接把较矮侧高度赋给 `h`，天然就是两边的
最小值，所以完全不需要 `MIN` 宏。代码更干净，也少了一次比较。

**(d) 正确性论证 -- 跳过不会遗漏最优解**

假设当前较矮侧高度为 `h`，宽度为 `w = right - left`。被跳过的任意位置高度
`h' <= h`，对应宽度 `w' < w`（因为指针往里移动了）。该位置能构成的面积上界为
`h' * w' <= h * w' < h * w`。而 `h * w` 要么就是当前面积（已经和 `max_area`
比较过了），要么比 `max_area` 更小。所以被跳过的位置绝不可能产生比 `max_area`
更大的面积，跳过是安全的。

## 复杂度分析

| 指标 | 复杂度 | 说明 |
|------|--------|------|
| 时间 | O(n) | 每个元素最多被 `left` 或 `right` 访问一次，两个指针总共最多移动 n-1 步 |
| 空间 | O(1) | 只使用了 `left`、`right`、`h`、`max_area` 四个变量 |

## 与原始解法对比

| | 原始版本 | 优化版本 |
|--|---------|---------|
| 指针移动 | 每次 +1 | 跳过所有 <= 当前 h 的线 |
| 面积计算 | 每步一次 | 每个唯一高度级别一次 |
| MIN 宏 | 使用 | 不需要 |
| 最坏时间 | O(n) | O(n) |
| 最优迭代 | n-1 | 可能远少于 n-1 |
| 空间 | O(1) | O(1) |

## 总结

优化版本的核心洞察非常简单：如果你已经知道较矮一侧的高度是 `h`，那么所有高度
`<= h` 的线都不值得停下来计算面积，因为宽度在缩小而高度没有增加，面积只会更小。
所以用一个内层 `while` 把它们全部跳过，直到遇到一条严格更高的线。这不改变算法的
渐近复杂度（仍然是 O(n) 时间、O(1) 空间），但在实际输入上可以大幅减少外层循环
的迭代次数和面积计算次数。以经典示例 `[1,8,6,2,5,4,8,3,7]` 为例，原始版本需要
8 次外层迭代，优化版本只需要 3 次。同时，由于 `if-else` 分支已经确定了较矮侧，
`MIN` 宏也自然消除了，代码更加简洁清晰。
